## 2. EXECUTION ORDER (RATIONALE)

The order optimizes for: (a) safety of each step, (b) ability to verify in isolation, (c) blocking dependencies.

### Stage A — Foundation (NO production behavior change)
1. **PRE: Migration discipline** — set up `packages/db/migrations/`, baseline snapshot, runner script, CI block on `db:push`. No DB changes yet.

### Stage B — Stop the bleeding (silent prod failures)
2. **Fix 13a: Make worker run** — add `@Cron` for `processQueue`, but with a kill-switch env var so we can disable without redeploy if it explodes.
3. **Fix 1: Idempotency on `automation_queue`** — `triggerKey` column + unique index + `onConflictDoNothing`. Without this, Fix 13a turns a backlog into duplicate sends.
4. **Fix 13b: Watchdog** — flip `processing > 10min` rows back to `pending`. Combined with Fix 1's idempotency, retried actions are safe.
5. **Fix 9: Idempotency tokens on outbound** — defence-in-depth. Even if our queue dedupes, the external API is the final stop. WapiSnap, GBP, OpenRouter.

### Stage C — Escalation hardening
6. **Fix 3: Async escalation evaluation** — wrap the call in an internal job. Decision point: reuse `automation_queue` with a new actionType, or add `internal_jobs` table per spec. Recommendation: `internal_jobs` (cleaner separation).

8. **Fix 5: SLA pause** — schema additions, enum value, methods, watcher update.
9. **Fix 11: Routing fallback** — owner-fallback + notification on routing failure.
10. **Fix 12: Bulk operations** — additive API, no migration.

### Stage D — Customer-experience guardrails

12. **Fix 8: Customer rate caps** — workspace settings + worker check.

### Stage E — Public-safety final boss

### Stage F — Verification

## 4. RISKS / UNRESOLVED OPEN QUESTIONS

These need user-confirmation before I touch the code:

### R1. WapiSnap idempotency support — UNVERIFIED
The bridge service at `apps/api/src/wapisnap/wapisnap-client.service.ts` uses signature headers (`X-Bridge-Timestamp`, `X-Bridge-Signature`, `X-Api-Key`). Whether the WapiSnap bridge **accepts** an `X-Idempotency-Key` header and **honours it** (returns same response for same key) is **not documented in the codebase**.

**Decision needed:** ask WapiSnap maintainers, OR ship Fix 9 with WapiSnap's idempotency as best-effort (we send the header; if their bridge ignores it, we still rely on Fix 1's queue-level dedup as the primary defense).

**My recommendation:** ship best-effort. Document in code comment that bridge support is unverified.

### R2. GBP API doesn't natively support idempotency keys
Confirmed: Google's My Business API (v4) for `reviews.reply` is a `PUT` that's already idempotent at the resource level (replying twice with the same content is a no-op). For `localPosts` it's a `POST` and *not* idempotent. Spec calls for "check before posting" pattern.

**Decision needed:** for `replyToReview`, do nothing (PUT is idempotent). For posts, implement read-before-write. OK with me?

**My recommendation:** yes, exactly that.

### R3. `escalation_status` enum has `closed` value — spec doesn't list it
Audit confirmed enum is `['open', 'in_progress', 'resolved', 'expired', 'closed']`. PRD only lists 4 (no `closed`). Either the spec has a gap or `closed` is legacy / additional.

**Decision:** keep `closed`. It's already in production. Fix 5 just adds `paused`.

### R4. Internal jobs vs. reuse automation_queue for Fix 3
Spec proposes a new `internal_jobs` table. Alternative: reuse `automation_queue` with a new internal `actionType` like `evaluate_escalation`.

**My recommendation:** stick with spec — separate `internal_jobs` table. Reasons:
- `automation_queue` is workspace/customer/review-keyed; internal jobs are system-keyed
- `internal_jobs` has different retry semantics (3 attempts, fixed backoff) vs `automation_queue` (per-action retry policy)
- Cleaner mental model; `internal_jobs` is for system-internal async work

### R5. Test coverage requirement
Spec mandates unit + integration tests per fix. Current codebase has **one** test file (the one I added — `journey-metrics.test.ts`). No vitest config beyond what I set up in `packages/shared`.

**Decision needed:** do you want me to:
- **(a)** stand up real test infrastructure (vitest config in `apps/api`, test DB setup, CI integration) and write tests per spec
- **(b)** write tests as `.test.ts` colocated with code, runnable via `npx vitest`, but skip CI integration
- **(c)** skip tests for Phase 0 — ship with manual verification only

**My recommendation:** **(b)**. Real test infra (a) doubles the timeline and is its own project. (c) breaks the spec's discipline and risks regressions. (b) gets us tests where it matters most without yak-shaving.

### R6. Production deploy cadence
Each fix is one PR per spec. With ~12 PRs ahead, do we:
- **(a)** Merge & deploy each PR individually as it lands (12 deploys)
- **(b)** Merge each PR but batch-deploy at logical milestones (Stage B → 1 deploy, Stage C → 1 deploy, etc., 5 deploys)
- **(c)** Merge all into a `phase-0/safety-fixes` branch, then 1 mega-deploy at end

**My recommendation:** **(b)**. Single deploys are too noisy; mega-deploy is too risky. Stage-grouped deploys give us 5 verifiable rollback points.

### R7. CI/CD currently?
The audit didn't surface a CI config. Is there a `.github/workflows/` folder, or is deploy purely DigitalOcean App Platform's auto-build on `main` push?

**Decision needed:** confirm — if no CI, "block `db:push` in production" must be enforced via runtime check in the migration runner (not CI).
